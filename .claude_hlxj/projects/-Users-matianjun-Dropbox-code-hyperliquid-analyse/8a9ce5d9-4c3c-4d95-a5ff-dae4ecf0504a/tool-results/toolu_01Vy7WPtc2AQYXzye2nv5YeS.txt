    98→## 设计动机
    99→
   100→### 原有架构的问题
   101→
   102→原 `OnlineTradingV3` 是一个单体架构，存在以下问题：
   103→
   104→| 问题 | 描述 | 影响 |
   105→|------|------|------|
   106→| **耦合度高** | 计算和执行逻辑在同一进程 | 难以独立扩展和部署 |
   107→| **单点故障** | 任一环节失败导致整体失败 | 系统可用性低 |
   108→| **重启丢失** | 进程重启后丢失当前周期的目标 | 踏空交易机会 |
   109→| **难以测试** | 计算和执行逻辑混合 | 单元测试复杂 |
   110→
   111→### 新架构的优势
   112→
   113→| 优势 | 描述 |
   114→|------|------|
   115→| **物理分离** | Ranker 和 Executor 可独立部署在不同服务器 |
   116→| **故障隔离** | Executor 崩溃不影响 Ranker 计算 |
   117→| **状态恢复** | Pull-on-Start 机制确保重启后不丢失信号 |
   118→| **水平扩展** | 可部署多个 Executor 对应不同交易所 |
   119→| **易于测试** | 通过 Mock Redis 即可完整测试通信流程 |

   123→## 通信协议
   124→
   125→### Last Value Cache + Signal 模式
   126→
   127→我们采用 **"Redis SET 存状态 + Redis PUBLISH 发信号"** 的模式，而非单纯的 Pub/Sub 数据传输。
   128→
   129→#### 为什么不用纯 Pub/Sub？
   130→
   131→| 方案 | 问题 |
   132→|------|------|
   133→| 纯 Pub/Sub | 如果 Executor 在 PUBLISH 时未订阅，消息丢失 |
   134→| 纯 Polling | 频繁轮询造成资源浪费 |
   135→| **Last Value Cache + Signal** | 数据持久化 + 实时通知，兼顾可靠性和实时性 |
   136→
   137→### Redis 操作序列
   138→
   139→#### Ranker (生产者) 操作顺序
   140→
   141→```python
   142→# ⚠️ 顺序至关重要：必须先 SET 后 PUBLISH
   143→# Step 1: SET - 存储目标权重 (Last Value Cache)
   144→await redis.set(f"strategy:target_weights:{account_id}", payload.to_json())
   145→
   146→# Step 2: PUBLISH - 发送信号通知 Executor
   147→await redis.publish(f"signal:rebalance:{account_id}", "update")
   148→```
   149→
   150→#### Executor (消费者) 操作
   151→
   152→```python
   153→# 启动阶段 (Pull-on-Start)
   154→data = await redis.get(f"strategy:target_weights:{account_id}")
   155→if data:
   156→    execute_rebalance(data)
   157→
   158→# 运行阶段 (Subscribe Loop)
   159→async for message in pubsub.listen():
   160→    if message["data"] == "update":
   161→        data = await redis.get(f"strategy:target_weights:{account_id}")
   162→        execute_rebalance(data)
   163→```
   164→
   165→### Redis Key 规范
   166→
   167→| 类型 | Key 格式 | 示例 |
   168→|------|----------|------|
   169→| 存储键 | `strategy:target_weights:{account_id}` | `strategy:target_weights:1` |
   170→| 信号频道 | `signal:rebalance:{account_id}` | `signal:rebalance:1` |
   171→| 信号消息 | `"update"` | 固定字符串 |

   218→## 模块详解
   219→
   220→### RankerProcess (online/ranker_process.py)
   221→
   222→**职责**：接收市场数据、计算特征、模型预测、发布目标权重
   223→
   224→#### 提前触发预测机制
   225→
   226→Ranker 采用**提前 1 分钟触发**策略，在 K 线关闭前就开始预测和数据处理，以减少延迟。
   227→
   228→##### 触发时间
   229→
   230→| K线周期 | K线关闭时间 | 预测触发时间 | 提前量 |
   231→|---------|-------------|--------------|--------|
   232→| 00:00-00:15 | 00:15:00 | **00:14:xx** | 1 分钟 |
   233→| 00:15-00:30 | 00:30:00 | **00:29:xx** | 1 分钟 |
   234→| 00:30-00:45 | 00:45:00 | **00:44:xx** | 1 分钟 |
   235→| 00:45-01:00 | 01:00:00 | **00:59:xx** | 1 分钟 |
   236→
   237→##### 触发条件
   238→
   239→```python
   240→if (
   241→    current_time.minute % 15 == 14           # 条件1: 14/29/44/59 分
   242→    and not self.prediction_triggered_this_cycle  # 条件2: 本周期未触发过
   243→    and self.all_symbol_updated              # 条件3: 所有symbol都更新了
   244→):
   245→```
   246→
   247→| 条件 | 说明 |
   248→|------|------|
   249→| `minute % 15 == 14` | 分钟数对 15 取余等于 14，即 14/29/44/59 分 |
   250→| `prediction_triggered_this_cycle` | 防止同一分钟内重复触发 |
   251→| `all_symbol_updated` | 确保所有交易对都收到了当前周期的数据 |
   252→
   253→##### 周期状态管理
   254→
   255→```
   256→00:00:05 → reset_cycle_flags() 重置所有标志
   257→           - kline_updated = {sym: False, ...}
   258→           - prediction_triggered_this_cycle = False
   259→           - data_appended_this_cycle = {sym: False, ...}
   260→
   261→00:01~13 → 收到各 symbol 的 K线更新
   262→           - last_kline[symbol] = 最新数据
   263→           - kline_updated[symbol] = True
   264→
   265→00:14:01 → 触发条件满足，开始预测
   266→           - 将 last_kline 追加到 self.data
   267→           - 调用 _process_weights()
   268→           - 发布到 Redis
   269→           - prediction_triggered_this_cycle = True
   270→
   271→00:15:00 → K线关闭
   272→           - 更新 self.data 中的数据（非追加）
   273→           - kline_closed[symbol] = True
   274→
   275→00:15:05 → 进入新周期，重置标志
   276→```
   277→
   278→##### 数据同步逻辑
   279→
   280→由于预测在 K 线关闭前触发，需要处理数据的追加和更新：
   281→
   282→```python
   283→# 提前预测时：追加 last_kline 到 self.data
   284→if not self.data_appended_this_cycle.get(sym, False):
   285→    self.data[sym] = pd.concat([current_df, new_df], ignore_index=True)
   286→    self.data_appended_this_cycle[sym] = True
   287→
   288→# K线关闭时：更新已追加的数据（最终价格可能有变化）
   289→if self.data_appended_this_cycle.get(symbol, False):
   290→    # 更新最后一行
   291→    last_idx = len(self.data[symbol]) - 1
   292→    for col in ["open", "close", "high", "low", "volume"]:
   293→        self.data[symbol].loc[last_idx, col] = new_kline_data[col]
   294→else:
   295→    # 正常追加
   296→    self.data[symbol] = pd.concat([current_df, new_df], ignore_index=True)
   297→```
   298→
   299→#### 核心流程
   300→
   301→```
   302→start_websocket()
   303→    │
   304→    ├──► 连接 Binance WebSocket
   305→    │
   306→    ├──► 接收 15m K线数据
   307→    │       │
   308→    │       ├──► 更新 last_kline[symbol]
   309→    │       └──► 标记 kline_updated[symbol] = True
   310→    │
   311→    ├──► 在 14/29/44/59 分 (提前1分钟触发):
   312→    │       │
   313→    │       ├──► 将 last_kline 追加到 self.data
   314→    │       │
   315→    │       ├──► _process_weights()
   316→    │       │       ├── 特征工程 (RankerFeature)
   317→    │       │       ├── 模型预测 (LightGBM)
   318→    │       │       └── 组合优化 (RegressionPortfolioOptimizer)
   319→    │       │
   320→    │       └──► _publish_target_weights()
   321→    │               ├── SET redis_key payload
   322→    │               └── PUBLISH redis_channel "update"
   323→    │
   324→    ├──► K线关闭时: 更新 self.data 中的最终数据
   325→    │
   326→    └──► 新周期开始 (0/15/30/45分): 重置状态
   327→```

   340→### ExecutorProcess (online/executor_process.py)
   341→
   342→**职责**：订阅信号、获取目标权重、执行交易
   343→
   344→#### 核心流程
   345→
   346→```
   347→start()
   348→    │
   349→    ├──► _pull_on_start()  # 启动即拉取
   350→    │       └── GET redis_key
   351→    │           └── _execute_rebalance() if data exists
   352→    │
   353→    └──► _subscribe_loop()  # 信号订阅循环
   354→            │
   355→            └──► for message in pubsub.listen():
   356→                    │
   357→                    └── if message == "update":
   358→                            ├── GET redis_key
   359→                            └── _execute_rebalance()
   360→```
   361→
   362→#### 关键特性
   363→
   364→| 特性 | 说明 |
   365→|------|------|
   366→| **Pull-on-Start** | 启动时主动拉取，防止重启踏空 |
   367→| **去重机制** | 基于 timestamp 去重，避免重复执行 |
   368→| **优雅降级** | Redis 连接失败时自动重试 |

   690→## 故障恢复
   691→
   692→### 场景 1: Executor 重启
   693→
   694→```
   695→时间线:
   696→  T0: Ranker 发布信号 (SET + PUBLISH)
   697→  T1: Executor 崩溃
   698→  T2: Executor 重启
   699→  T3: Pull-on-Start 执行 GET
   700→  T4: 获取 T0 的权重数据
   701→  T5: 执行交易 ✓
   702→```
   703→
   704→**结论**：不会丢失信号，因为数据存储在 Redis 中。
   705→
   706→### 场景 2: Ranker 重启
   707→
   708→```
   709→时间线:
   710→  T0: Ranker 崩溃
   711→  T1: Ranker 重启
   712→  T2: 等待下一个 15m K线周期
   713→  T3: 正常发布新信号
   714→```
   715→
   716→**结论**：最多丢失一个周期的信号。
   717→
   718→### 场景 3: Redis 连接失败
   719→
   720→```python
   721→# RankerProcess 中的错误处理
   722→try:
   723→    await redis.set(self.redis_key, payload.to_json())
   724→    await redis.publish(self.redis_channel, "update")
   725→except Exception as e:
   726→    logger.error(f"Failed to publish to Redis: {e}")
   727→    return False  # 返回失败状态，不会导致进程崩溃
   728→```
   729→
   730→**结论**：记录错误日志，等待下一周期重试。
