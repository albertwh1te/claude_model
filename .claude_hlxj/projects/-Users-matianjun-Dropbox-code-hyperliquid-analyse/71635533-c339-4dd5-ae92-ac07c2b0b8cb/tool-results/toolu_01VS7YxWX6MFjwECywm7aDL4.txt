     5→# Executes
     6→
     7→## Overview
     8→
     9→All executes go through the following endpoint; the exact details of the execution are specified by the JSON payload.
    10→
    11→* **Websocket**: <mark style="color:orange;">`WEBSOCKET [GATEWAY_WEBSOCKET_ENDPOINT]`</mark>
    12→* **REST**: <mark style="color:orange;">`POST [GATEWAY_REST_ENDPOINT]/execute`</mark>
    13→
    14→### **Signing**
    15→
    16→All executes are signed using [EIP712](https://eips.ethereum.org/EIPS/eip-712). Each execute request contains:
    17→
    18→1. A piece of structured data that includes the sender address
    19→2. A signature of the hash of that structured data, signed by the sender
    20→
    21→You can check the SDK for some examples of how to generate these signatures.
    22→
    23→{% hint style="info" %}
    24→See more info in the [signing](https://docs.nado.xyz/developer-resources/api/gateway/signing) page.
    25→{% endhint %}
    26→
    27→### **Sender Field Structure**
    28→
    29→The sender field is a solidity <mark style="color:red;">`bytes32`</mark> . There are two components:
    30→
    31→* an <mark style="color:red;">`address`</mark> that is a <mark style="color:red;">`bytes20`</mark>
    32→* a subaccount identifier that is a <mark style="color:red;">`bytes12`</mark>
    33→
    34→For example, if your address was <mark style="color:red;">`0x7a5ec2748e9065794491a8d29dcf3f9edb8d7c43`</mark>, and you wanted to use the default subaccount identifier (i.e: the word <mark style="color:red;">`default`</mark>) you can set <mark style="color:red;">`sender`</mark> to <mark style="color:red;">`0x7a5ec2748e9065794491a8d29dcf3f9edb8d7c4364656661756c740000000000`</mark> , which sets the subaccount identifier to <mark style="color:red;">`64656661756c740000000000`</mark>.
    35→
    36→### **Amounts**
    37→
    38→For <mark style="color:red;">`DepositCollateral`</mark> and <mark style="color:red;">`WithdrawCollateral`</mark>, the amount specifies the physical token amount that you want to receive. `i.e.` if USDT0 has 6 decimals, and you want to deposit or withdraw 1 USDT0, you specify <mark style="color:red;">`amount = 1e6`</mark>.
    39→
    40→For all other transactions, amount is normalized to 18 decimals, so <mark style="color:red;">`1e18`</mark> == one unit of the underlying asset. For example, if you want to buy 1 wETH, regardless of the amount of decimals the wETH contract has on chain, you specify <mark style="color:red;">`1e18`</mark> in the amount field of the order.
    41→
    42→## API Response
    43→
    44→All `Execute` messages return the following information:
    45→
    46→#### Success
    47→
    48→```json
    49→{
    50→  "status": "success",
    51→  "signature": "{signature}",
    52→  "data"?: {data_obj},
    53→  "request_type": "{request_type}"
    54→}
    55→```
    56→
    57→#### Failure
    58→
    59→```json
    60→{
    61→  "status": "failure",
    62→  "signature": "{signature}",
    63→  "error": "{error_msg}",
    64→  "error_code": {error_code},
    65→  "request_type": "{request_type}"
    66→}
    67→```

   260→## Signing
   261→
   262→{% hint style="info" %}
   263→See more details and examples in our [signing](https://docs.nado.xyz/developer-resources/api/gateway/signing) page.
   264→{% endhint %}
   265→
   266→The solidity typed data struct that needs to be signed is:
   267→
   268→```solidity
   269→struct CancellationProducts {
   270→    bytes32 sender;
   271→    uint32[] productIds;
   272→    uint64 nonce;
   273→}
   274→```
   275→
   276→<mark style="color:red;">`sender`</mark>: a <mark style="color:red;">`bytes32`</mark> sent as a hex string; includes the address and the subaccount identifier
   277→
   278→<mark style="color:red;">`productIds`</mark>: a list of product Ids for which to cancel all subaccount orders. When left empty, orders from all products will be cancelled.
   279→
   280→<mark style="color:red;">`nonce`</mark>: used to differentiate between the same cancellation multiple times, and a user trying to place a cancellation with the same parameters twice. Sent as a string. Encodes two bit of information:
   281→
   282→* Most significant <mark style="color:red;">`44`</mark> bits encoding the <mark style="color:red;">`recv_time`</mark> in milliseconds after which the cancellation should be ignored by the matching engine; the engine will accept cancellations where <mark style="color:red;">`current_time < recv_time <= current_time + 100000`</mark>
   283→* Least significant <mark style="color:red;">`20`</mark> bits are a random integer used to avoid hash collisions
   284→
   285→  For example, to place a cancellation with a random integer of <mark style="color:red;">`1000`</mark>, and a discard time 50 ms from now, we would send a nonce of <mark style="color:red;">`(timestamp_ms() + 50) << 20 + 1000`</mark>
   286→
   287→{% hint style="warning" %}
   288→**Note**: for signing you should always use the data type specified in the solidity struct which might be different from the type sent in the request e.g: <mark style="color:red;">`nonce`</mark> should be an <mark style="color:red;">`uint64`</mark> for **Signing** but should be sent as a <mark style="color:red;">`string`</mark> in the final payload.
   289→{% endhint %}

   617→## Rate limits
   618→
   619→The rate limit weight varies based on the request parameters:
   620→
   621→* **Basic query** (no `txns`): **weight = 2**
   622→  * 1200 requests/min or 200 requests every 10 seconds per IP address
   623→* **With simulation** (`txns` provided): **weight = 10**
   624→  * 240 requests/min or 40 requests every 10 seconds per IP address
   625→* **With simulation + pre\_state** (`txns` and `pre_state="true"`): **weight = 15**
   626→  * 160 requests/min or \~26 requests every 10 seconds per IP address
   627→
   628→{% hint style="info" %}

   675→

   683→
   684→Updates internal balances for the <mark style="color:red;">`product_id`</mark> and amount deltas provided.

   697→## Response
   698→
   699→{% hint style="info" %}
   700→**Note**:
   701→
   702→* <mark style="color:red;">`healths`</mark>:
   703→  * <mark style="color:red;">`healths[0]`</mark>: info about your initial health, which is weighted by `long_weight_initial_x18` and `short_weight_initial_x18.`
   704→  * <mark style="color:red;">`healths[1]`</mark>: info about your maintenance health, which is weighted by `long_weight_maintenance_x18` and `short_weight_maintenance_x18.`
   705→  * <mark style="color:red;">`healths[2]`</mark>: info about your unweighted health.
   706→* <mark style="color:red;">`health_contributions`</mark> is indexed by <mark style="color:red;">product\_id</mark> and represents the contribution of the corresponding product to the final health.
   707→  * <mark style="color:red;">`health_contributions[product_id][0]`</mark>`: contribution to healths[0]`
   708→  * <mark style="color:red;">`health_contributions[product_id][1]`</mark>`: contribution to healths[1]`
   709→  * <mark style="color:red;">`health_contributions[product_id][2]`</mark>`: contribution to healths[2]`

   713→  * <mark style="color:red;">`pre_state.spot_balances`</mark>: Spot balances before transactions

  1101→# Signing
  1102→
  1103→All executes are signed using [EIP712](https://eips.ethereum.org/EIPS/eip-712). Each execute request contains:
  1104→
  1105→1. A piece of structured data that includes the sender address i.e: the <mark style="color:red;">`primaryType`</mark> that needs to be signed.
  1106→2. A signature of the hash of that structured data, signed by the sender.
  1107→
  1108→## Domain
  1109→
  1110→The following is the domain required as part of the EIP712 structure:
  1111→
  1112→```json
  1113→{
  1114→    name: 'Nado',
  1115→    version: '0.0.1',
  1116→    chainId: chainId,
  1117→    verifyingContract: contractAddress
  1118→}
  1119→```
  1120→
  1121→You can retrieve the corresponding chain id and verifying contract via the [contracts](https://docs.nado.xyz/developer-resources/api/gateway/queries/contracts) query.
  1122→
  1123→{% hint style="warning" %}
  1124→**Note**: make sure to use the correct verifying contract for each execute:
  1125→
  1126→* For place order: should use `address(producId)` i.e: the 20 bytes hex representation of the `productId` for the order. For example, the verify contract of product `18` is `0x0000000000000000000000000000000000000012` .
  1127→* For everything else: should use the endpoint address.
  1128→
  1129→See more details in the [contracts](https://docs.nado.xyz/developer-resources/api/gateway/queries/contracts) query page.
  1130→{% endhint %}
  1131→
  1132→```python
  1133→def gen_order_verifying_contract(product_id: int) -> str:
  1134→    """
  1135→    Generates the order verifying contract address based on the product ID.
  1136→
  1137→    Args:
  1138→        product_id (int): The product ID for which to generate the verifying contract address.
  1139→
  1140→    Returns:
  1141→        str: The generated order verifying contract address in hexadecimal format.
  1142→    """
  1143→    be_bytes = product_id.to_bytes(20, byteorder="big", signed=False)
  1144→    return "0x" + be_bytes.hex()
  1145→```

  1427→# Examples
  1428→
  1429→The following are full examples of EIP12 typed data for each of Nado's executes. Each execute includes a <mark style="color:red;">`sender`</mark> field which is a solidity <mark style="color:red;">`bytes32`</mark> . There are two components to this field:
  1430→
  1431→* an <mark style="color:red;">`address`</mark> that is a <mark style="color:red;">`bytes20`</mark>
  1432→* a subaccount identifier that is a <mark style="color:red;">`bytes12`</mark>
  1433→
  1434→For example, if your address was <mark style="color:red;">`0x7a5ec2748e9065794491a8d29dcf3f9edb8d7c43`</mark>, and you wanted to use the default subaccount identifier (i.e: an empty identifier `""`) you can set <mark style="color:red;">`sender`</mark> to <mark style="color:red;">`0x7a5ec2748e9065794491a8d29dcf3f9edb8d7c43000000000000000000000000`</mark> , which sets all bytes of the subaccount identifier to <mark style="color:red;">`0`</mark>.
  1435→
  1436→{% hint style="info" %}
  1437→**Note**: a <mark style="color:red;">`bytes32`</mark> representation of the sender must used when signing the request.
  1438→{% endhint %}
  1439→
  1440→See below a sample util to convert a hex to a **bytes32**:
  1441→
  1442→{% tabs %}
  1443→{% tab title="Python" %}
  1444→
  1445→```python
  1446→def hex_to_bytes32(hex_string):
  1447→    if hex_string.startswith("0x"):
  1448→        hex_string = hex_string[2:]
  1449→    data_bytes = bytes.fromhex(hex_string)
  1450→    padded_data = data_bytes + b"\x00" * (32 - len(data_bytes))
  1451→    return padded_data
  1452→
  1453→sender = hex_to_bytes32('0x841fe4876763357975d60da128d8a54bb045d76a64656661756c740000000000')
  1454→```
  1455→
  1456→{% endtab %}
  1457→
  1458→{% tab title="Typescript" %}
  1459→
  1460→```typescript
  1461→import { arrayify } from 'ethers/lib/utils';
  1462→
  1463→export function hexToBytes32(subaccount: string) {
  1464→  const subaccountBytes = arrayify(subaccount);
  1465→  const bytes32 = new Uint8Array(32);
  1466→  for (let i = 0; i < Math.min(subaccountBytes.length, 32); i++) {
  1467→    bytes32[i] = subaccountBytes[i];
  1468→  }
  1469→  return bytes32;
  1470→}
  1471→
  1472→const sender = hexToBytes32('0x841fe4876763357975d60da128d8a54bb045d76a64656661756c740000000000')
  1473→```

  1804→# Q\&A
  1805→
  1806→### Q: **What is Nado's EIP712 domain?**
  1807→
  1808→```json
  1809→{
  1810→    name: 'Nado',
  1811→    version: '0.0.1',
  1812→    chainId: chainId,
  1813→    verifyingContract: contractAddress
  1814→}
  1815→```
  1816→
  1817→{% hint style="info" %}
  1818→See [signing](https://docs.nado.xyz/developer-resources/api/gateway/signing/..#domain) for more details.
  1819→{% endhint %}
  1820→
  1821→### Q: How can i retrieve the verifying contracts to use?
  1822→
  1823→* Via the [contracts](https://docs.nado.xyz/developer-resources/api/gateway/queries/contracts) query for all executes except place orders.
  1824→
  1825→### Q: Which contract should I use for each execute?
  1826→
  1827→* For place orders: must be computed as <mark style="color:red;">`address(productId)`</mark>. For example, the verify contract of product <mark style="color:red;">`18`</mark> is <mark style="color:red;">`0x0000000000000000000000000000000000000012`</mark>.
  1828→* For everything else: use the endpoint contract from the contracts query.
  1829→
  1830→{% hint style="info" %}
  1831→See the [contracts](https://docs.nado.xyz/developer-resources/api/queries/contracts#response) query for more details.
  1832→{% endhint %}
  1833→
  1834→### Q: I am running into signature errors, how to fix?
  1835→
  1836→Signature errors can arise for several reasons:
  1837→
  1838→* **An invalid struct**: confirm you are signing the correct struct. See the [Signing](https://docs.nado.xyz/developer-resources/api/gateway/signing) page to verify the struct of each execute request.
  1839→* **An invalid chain id**: confirm you have the correct chain id for the network you are on.
  1840→* **An invalid verifying contract**: confirm you have the correct verifying contract address for the network and execute you are signing. i.e: confirm you are using the correct orderbook address for place orders and endpoint address for everything else.
  1841→
  1842→### Q: Is any other signing standard supported?
  1843→
  1844→No, only [EIP712](https://eips.ethereum.org/EIPS/eip-712).
  1845→
  1846→### Q: Are there any examples you can provide?
  1847→
  1848→See [examples](https://docs.nado.xyz/developer-resources/api/gateway/signing/examples).
  1849→
  1850→### Q: What is the PrimaryType of execute X?
  1851→
  1852→All primary types are listed in our [signing](https://docs.nado.xyz/developer-resources/api/gateway/signing) page.

  1883→# Rate limits
  1884→
  1885→## Overview
  1886→
  1887→* Nado uses a weight-based rate-limiting system across queries and executes. We limit based on <mark style="color:red;">`IP address`</mark>, <mark style="color:red;">`Wallet address`</mark>, and a global <mark style="color:red;">`max # of orders per subaccount per market`</mark>.
  1888→* These limits equally apply to both <mark style="color:red;">`http`</mark> requests and <mark style="color:red;">`Websocket`</mark> messages.
  1889→* Limits are applied on a <mark style="color:red;">`1 minute`</mark> and <mark style="color:red;">`10 seconds`</mark> basis.
  1890→
  1891→## Limits
  1892→
  1893→* IP addresses have a max weight limit of <mark style="color:red;">`2400`</mark> per minute or <mark style="color:red;">`400`</mark> every 10 seconds applied only to queries.
  1894→* Wallet addresses have a max weight limit of <mark style="color:red;">`600`</mark> per minute or <mark style="color:red;">`100`</mark> every 10 seconds applied only to executes.
  1895→* Users can have up to <mark style="color:red;">`500`</mark> open orders per subaccount per market.
  1896→* Orders have the following additional limits:
  1897→  * Place orders (with spot leverage): up to <mark style="color:red;">`600`</mark> per minute or <mark style="color:red;">`100`</mark> every 10 seconds across all markets.
  1898→  * Place orders (without spot leverage): up to <mark style="color:red;">`30`</mark> per minute or <mark style="color:red;">`5`</mark> every 10 seconds across all markets. **Note**: orders without spot leverage are <mark style="color:red;">`20x`</mark> more expensive to place due to additional health checks needed.
  1899→  * Order cancellations: up to <mark style="color:red;">`600`</mark> per minute or <mark style="color:red;">`100`</mark> every 10 seconds.
  1900→
  1901→## Query Weights
  1902→
  1903→Queries are rate-limited based on IP. The following weights are applied per query:
  1904→
  1905→* [**Status**](https://docs.nado.xyz/developer-resources/api/gateway/queries/status): <mark style="color:red;">`IP weight = 1`</mark>
  1906→* [**Contracts**](https://docs.nado.xyz/developer-resources/api/gateway/queries/contracts): <mark style="color:red;">`IP weight = 1`</mark>
  1907→* [**Nonces**](https://docs.nado.xyz/developer-resources/api/gateway/queries/nonces): <mark style="color:red;">`IP weight = 2`</mark>
  1908→* [**Order**](https://docs.nado.xyz/developer-resources/api/gateway/queries/order)**:** <mark style="color:red;">`IP weight = 1`</mark>
  1909→* [**Orders**](https://docs.nado.xyz/developer-resources/api/gateway/queries/orders): <mark style="color:red;">`IP weight = 2 * product_ids.length`</mark>
  1910→* [**Subaccount Info**](https://docs.nado.xyz/developer-resources/api/gateway/queries/subaccount-info): <mark style="color:red;">`IP weight = 2`</mark> (or <mark style="color:red;">`10`</mark> with `txns`, or <mark style="color:red;">`15`</mark> with `txns` + `pre_state="true"`)
  1911→* [**Isolated Positions**](https://docs.nado.xyz/developer-resources/api/gateway/queries/isolated-positions): <mark style="color:red;">`IP weight = 10`</mark>
  1912→* [**Market Liquidity**](https://docs.nado.xyz/developer-resources/api/gateway/queries/market-liquidity): <mark style="color:red;">`IP weight = 1`</mark>
  1913→* [**Symbols**](https://docs.nado.xyz/developer-resources/api/gateway/queries/symbols): <mark style="color:red;">`IP weight = 2`</mark>

  1930→## Archive (indexer) Weights
  1931→
  1932→* Archive (indexer) queries are rate-limited based on IP.
  1933→* IP addresses have a max weight limit of <mark style="color:red;">`2400`</mark> per minute or <mark style="color:red;">`400`</mark> every 10 seconds.
  1934→
  1935→The following weights are applied per query:
  1936→
  1937→* [**Orders**](https://docs.nado.xyz/developer-resources/api/archive-indexer/orders)**:** <mark style="color:red;">`IP Weight = 2 + (limit * subaccounts.length / 20)`</mark>; where <mark style="color:red;">`limit`</mark> and <mark style="color:red;">`subaccounts`</mark> are query params.
  1938→* [**Matches**](https://docs.nado.xyz/developer-resources/api/archive-indexer/matches)**:** <mark style="color:red;">`IP Weight = 2 + (limit * subaccounts.length / 10)`</mark>; where <mark style="color:red;">`limit`</mark> and <mark style="color:red;">`subaccounts`</mark> are query params.
  1939→* [**Events**](https://docs.nado.xyz/developer-resources/api/archive-indexer/events)**:** <mark style="color:red;">`IP Weight = 2 + (limit * subaccounts.length / 10)`</mark>; where <mark style="color:red;">`limit`</mark> and <mark style="color:red;">`subaccounts`</mark> are query params.
  1940→* [**Candlesticks**](https://docs.nado.xyz/developer-resources/api/archive-indexer/candlesticks)**:** <mark style="color:red;">`IP Weight = 1 + limit / 20`</mark>; where <mark style="color:red;">`limit`</mark> is a query param.
  1941→* [**Edge Candlesticks**](https://docs.nado.xyz/developer-resources/api/archive-indexer/edge-candlesticks): <mark style="color:red;">`IP Weight = 1 + limit / 20`</mark>; where <mark style="color:red;">`limit`</mark> is a query param.
  1942→* [**Product Snapshots**](https://docs.nado.xyz/developer-resources/api/archive-indexer/product-snapshots)**:** <mark style="color:red;">`IP Weight = 10`</mark> for single `products` query, or <mark style="color:red;">`10 * timestamps.length`</mark> for multiple `product_snapshots` query with max\_time parameter
  1943→* [**Funding Rate**](https://docs.nado.xyz/developer-resources/api/archive-indexer/funding-rate)**:** <mark style="color:red;">`IP Weight = 2`</mark>
  1944→* [**Interest & funding payments**](https://docs.nado.xyz/developer-resources/api/archive-indexer/interest-and-funding-payments)**:** <mark style="color:red;">`IP Weight = 5`</mark>
  1945→* [**Oracle Price**](https://docs.nado.xyz/developer-resources/api/archive-indexer/oracle-price)**:** <mark style="color:red;">`IP Weight = 2`</mark>
  1946→* [**Oracle Snapshots**](https://docs.nado.xyz/developer-resources/api/archive-indexer/oracle-snapshots): <mark style="color:red;">`IP Weight = max((snapshot_count * product_ids.length / 100), 2)`</mark>; where snapshot\_count is <mark style="color:red;">`interval.count.min(500)`</mark>
  1947→* [**Perp Prices**](https://docs.nado.xyz/developer-resources/api/archive-indexer/perp-prices)**:** <mark style="color:red;">`IP Weight = 2`</mark> (includes both single `price` and multiple `perp_prices` queries)
  1948→* [**Market Snapshots**](https://docs.nado.xyz/developer-resources/api/archive-indexer/market-snapshots)**:** <mark style="color:red;">`IP Weight = max((snapshot_count * product_ids.length / 100), 2)`</mark>; where snapshot\_count is <mark style="color:red;">`interval.count.min(500)`</mark>
  1949→* [**Edge Market Snapshots**](https://docs.nado.xyz/developer-resources/api/archive-indexer/edge-market-snapshots): <mark style="color:red;">`IP weight = (interval.count.min(500) / 20) + (interval.count.clamp(2, 20) * 2)`</mark>
  1950→* [**Subaccounts**](https://docs.nado.xyz/developer-resources/api/archive-indexer/subaccounts)**:** <mark style="color:red;">`IP Weight = 2`</mark>
  1951→* [**Subaccount Snapshots**](https://docs.nado.xyz/developer-resources/api/archive-indexer/subaccount-snapshots): <mark style="color:red;">`IP Weight = 2 + (limit * subaccounts.length / 10)`</mark>; where <mark style="color:red;">`limit`</mark> and <mark style="color:red;">`subaccounts`</mark> are query params.
  1952→* [**Linked Signers**](https://docs.nado.xyz/developer-resources/api/archive-indexer/linked-signers): <mark style="color:red;">`IP Weight = 2`</mark>
  1953→* [**Linked Signer Rate Limit**](https://docs.nado.xyz/developer-resources/api/archive-indexer/linked-signer-rate-limit)**:** <mark style="color:red;">`IP Weight = 2`</mark>
  1954→* [**Isolated Subaccounts**](https://docs.nado.xyz/developer-resources/api/archive-indexer/isolated-subaccounts): <mark style="color:red;">`IP Weight = 2`</mark>
  1955→* [**Signatures**](https://docs.nado.xyz/developer-resources/api/archive-indexer/signatures): <mark style="color:red;">`IP Weight = 2 + len(digests) / 10`</mark>; where <mark style="color:red;">`digests`</mark> is a query param.
  1956→* [**Fast Withdrawal Signature**](https://docs.nado.xyz/developer-resources/api/archive-indexer/fast-withdrawal-signature): <mark style="color:red;">`IP Weight = 10`</mark>
  1957→* [**NLP Funding Payments**](https://docs.nado.xyz/developer-resources/api/archive-indexer/nlp-funding-payments): <mark style="color:red;">`IP Weight = 5`</mark>
  1958→* [**NLP Interest Payments**](https://docs.nado.xyz/developer-resources/api/archive-indexer/nlp-interest-payments): <mark style="color:red;">`IP Weight = 5`</mark>
  1959→* [**NLP Snapshots**](https://docs.nado.xyz/developer-resources/api/archive-indexer/nlp-snapshots): <mark style="color:red;">`IP Weight = limit.min(500) / 100`</mark>; where <mark style="color:red;">`limit`</mark> is a query param.
  1960→* [**Tx Hashes**](https://docs.nado.xyz/developer-resources/api/archive-indexer/tx-hashes): <mark style="color:red;">`IP Weight = idxs.length * 2`</mark>; where <mark style="color:red;">`idxs`</mark> is an array of submission indices (max 100).
  1961→* [**Liquidation Feed**](https://docs.nado.xyz/developer-resources/api/archive-indexer/liquidation-feed)**:** <mark style="color:red;">`IP Weight = 2`</mark>
  1962→* [**Sequencer Backlog**](https://docs.nado.xyz/developer-resources/api/archive-indexer/sequencer-backlog): <mark style="color:red;">`IP Weight = 1`</mark>
  1963→* [**Direct Deposit Address**](https://docs.nado.xyz/developer-resources/api/archive-indexer/direct-deposit-address): <mark style="color:red;">`IP Weight = 10`</mark>
  1964→* [**Quote Price**](https://docs.nado.xyz/developer-resources/api/archive-indexer/quote-price): <mark style="color:red;">`IP Weight = 2`</mark>

  1967→## Execute Weights
  1968→
  1969→Executes are rate-limited based on Wallet address. The following weights are applied per execute:
  1970→
  1971→* [**Place order**](https://docs.nado.xyz/developer-resources/api/gateway/executes/place-order)**:**
  1972→  * With spot leverage: <mark style="color:red;">`Wallet weight = 1`</mark>
  1973→  * Without spot leverage: <mark style="color:red;">`Wallet weight = 20`</mark>
  1974→* [**Place orders**](https://docs.nado.xyz/developer-resources/api/gateway/executes/place-orders)**:**
  1975→  * With spot leverage: <mark style="color:red;">`Wallet weight = 1 per order`</mark>
  1976→  * Without spot leverage: <mark style="color:red;">`Wallet weight = 20 per order`</mark>
  1977→  * **Note**: 50ms processing penalty per request
  1978→* [**Cancel orders**](https://docs.nado.xyz/developer-resources/api/gateway/executes/cancel-orders)**:**
  1979→  * When no **digests** are provided: <mark style="color:red;">`Wallet weight = 1`</mark>
  1980→  * When **digests** are provided: <mark style="color:red;">`Wallet weight = total digests`</mark>
  1981→* [**Cancel Product Orders**](https://docs.nado.xyz/developer-resources/api/gateway/executes/cancel-product-orders)**:**
  1982→  * When no **productIds** are provide&#x64;**:** <mark style="color:red;">`Wallet weight = 50`</mark>
  1983→  * When **productIds** are provided: <mark style="color:red;">`Wallet weight = 5 * total productIds`</mark>
  1984→* [**Cancel And Place**](https://docs.nado.xyz/developer-resources/api/gateway/executes/cancel-and-place):
  1985→  * The sum of [Cancel orders](https://docs.nado.xyz/developer-resources/api/gateway/executes/cancel-orders) + [Place order](https://docs.nado.xyz/developer-resources/api/gateway/executes/place-order) limits
  1986→* [**Withdraw Collateral**](https://docs.nado.xyz/developer-resources/api/gateway/executes/withdraw-collateral)**:**
  1987→  * With spot leverage: <mark style="color:red;">`Wallet weight = 10`</mark>
  1988→  * Without spot leverage: <mark style="color:red;">`Wallet weight = 20`</mark>
  1989→* [**Liquidate Subaccount**](https://docs.nado.xyz/developer-resources/api/gateway/executes/liquidate-subaccount): <mark style="color:red;">`Wallet weight = 20`</mark>
  1990→* [**Mint NLP**](https://docs.nado.xyz/developer-resources/api/gateway/executes/mint-nlp): <mark style="color:red;">`Wallet weight = 10`</mark>
  1991→* [**Burn NLP**](https://docs.nado.xyz/developer-resources/api/gateway/executes/burn-nlp): <mark style="color:red;">`Wallet weight = 10`</mark>
  1992→* [**Link Signer**](https://docs.nado.xyz/developer-resources/api/gateway/executes/link-signer): <mark style="color:red;">`Wallet weight = 30`</mark>
  1993→  * Can only perform a max of 50 link signer requests every 7 days per subaccount.
  1994→* [**Transfer Quote:**](https://docs.nado.xyz/developer-resources/api/gateway/executes/transfer-quote) <mark style="color:red;">`Wallet weight = 10`</mark>
  1995→  * Can only transfer to a max of 5 new recipients within 24hrs.

  2005→# Definitions / Formulas
  2006→
  2007→## Definitions
  2008→
  2009→### **Unsettled USDT0**
  2010→
  2011→Perp balances have two main components:
  2012→
  2013→* <mark style="color:red;">`amount`</mark>
  2014→* <mark style="color:red;">`v_quote_balance`</mark>
  2015→
  2016→When you buy a perp, <mark style="color:red;">`amount`</mark> increments and <mark style="color:red;">`v_quote_balance`</mark> decrements, and vice versa for selling.
  2017→
  2018→Settlement is the process of converting from <mark style="color:red;">v\_quote\_balance</mark> into actual USDT0 balance. This happens mostly on position close, but may happen on extremely negative PNL positions when we need to pay out positive PNL positions.
  2019→
  2020→The amount that is transferred between <mark style="color:red;">`v_quote_balance`</mark> in the perp and your USDT0 balance is an amount that results in <mark style="color:red;">`amount * oracle_price + v_quote_balance == 0`</mark>. Unsettled USDT0 is the total amount that would be transferred between <mark style="color:red;">`v_quote_balance`</mark> and your USDT0 balance summed across all perps.
  2021→
  2022→### **Unsettled PNL**
  2023→
  2024→**Note:** Technically, there is no such concept as "Unsettled PNL" in our system. However, the UI displays "Unsettled PnL" in some places (e.g., in the USDT0 Balance section) for user clarity.
  2025→
  2026→**What the UI actually shows:** When you see "Unsettled PnL" in the UI, it refers to **Unsettled USDT0** (see above) - the total unsettled quote balance across all perp positions.
  2027→
  2028→**For developers:** Always use **Unsettled USDT0** when referring to this value programmatically. It represents the sum of <mark style="color:red;">`amount × oracle_price + v_quote_balance`</mark> across all perp positions, which is the amount that would be settled into your USDT0 balance.
  2029→
  2030→### **Unrealized PNL**
  2031→
  2032→Refers to the estimated gains or losses of a current position based on the difference between the average entry price and the current oracle price.
  2033→
  2034→## Formulas
  2035→
  2036→### **Unrealized PNL**
  2037→
  2038→Using the [indexer's events query](https://docs.nado.xyz/developer-resources/api/archive-indexer/events), your unrealized PNL at the end of some event is given by:
  2039→
  2040→{% code lineNumbers="true" %}
  2041→
  2042→```python
  2043→unrealized_pnl = (
  2044→    event.post_balance.amount * event.product.oracle_price_x18 
  2045→    - event.net_entry_unrealized
  2046→)
  2047→```
  2048→
  2049→{% endcode %}
  2050→
  2051→### Total PNL
  2052→
  2053→Your total PNL between <mark style="color:red;">`event1`</mark> and <mark style="color:red;">`event2`</mark>, assuming <mark style="color:red;">`event1`</mark> is after <mark style="color:red;">`event2`</mark> - is given by:
  2054→
  2055→<pre class="language-python" data-line-numbers><code class="lang-python"><strong>total_pnl = (
  2056→</strong><strong>    (event1.post_balance.amount * event1.product.oracle_price_x18 - event1.net_entry_cumulative)
  2057→</strong><strong>    - (event2.post_balance.amount * event2.product.oracle_price_x18 - event2.net_entry_cumulative)
  2058→</strong><strong>)
  2059→</strong></code></pre>
  2060→
  2061→{% hint style="info" %}
  2062→**Notes**:
  2063→
  2064→* You can use 0 for the second term for the PNL to compute since the beginning of time.
  2065→* For spots, we will count deposits and withdraws towards your PNL. i.e. if you deposit BTC, for PNL tracking purposes it is counted as a BTC long at the oracle price.
  2066→  {% endhint %}

  2151→# Order Appendix
  2152→
  2153→The **Order Appendix** is a 128-bit integer that encodes extra order parameters like execution type, isolated margin, and trigger configurations.
  2154→
  2155→## Bit Layout
  2156→
  2157→```json
  2158→| value   | reserved | trigger | reduce only | order type | isolated | version |
  2159→| 64 bits | 50 bits  | 2 bits  | 1 bit       | 2 bits     | 1 bit    | 8 bits  |
  2160→| 127..64 | 63..14   | 13..12  | 11          | 10..9      | 8        | 7..0    |
  2161→```
  2162→
  2163→## Fields (from LSB to MSB)
  2164→
  2165→### <mark style="color:red;">Version</mark>
  2166→
  2167→**8-bits (0-7)**. Protocol version identifier. Currently <mark style="color:red;">`1`</mark>. May increment when encoding structure updates.
  2168→
  2169→### <mark style="color:red;">Isolated</mark>
  2170→
  2171→**1-bit (8)**. Indicates whether the order uses isolated margin. Isolated positions have dedicated margin for a specific product, creating a separate isolated subaccount. The original account becomes the "parent subaccount" that can manage the isolated position.
  2172→
  2173→*Key Properties:*
  2174→
  2175→* Creates isolated subaccount with dedicated margin
  2176→* Only quote transfers allowed between isolated and parent subaccounts
  2177→* Parent account can sign orders for isolated subaccount
  2178→* Cannot be combined with TWAP orders
  2179→
  2180→*Example:*
  2181→
  2182→```python
  2183→from nado_protocol.utils.appendix import build_appendix
  2184→from nado_protocol.utils.math import to_x6
  2185→
  2186→# Create isolated order with 1000 USDT0 margin
  2187→appendix = build_appendix(
  2188→    order_type=OrderType.DEFAULT,
  2189→    isolated=True,
  2190→    isolated_margin=to_x6(1000)  # 1000 USDT0 (x6 precision)
  2191→)
  2192→```
  2193→
  2194→### <mark style="color:red;">Order Type</mark>
  2195→
  2196→**2-bits (9-10)**. Execution behavior for the order.
  2197→
  2198→*Values:*
  2199→
  2200→* <mark style="color:red;">`0`</mark> - <mark style="color:red;">`DEFAULT`</mark>: Standard limit order behavior.
  2201→* <mark style="color:red;">`1`</mark> - <mark style="color:red;">`IOC (Immediate or Cancel)`</mark>: Execute immediately, cancel unfilled portion.
  2202→* <mark style="color:red;">`2`</mark> - <mark style="color:red;">`FOK (Fill or Kill)`</mark>: Execute completely or cancel entire order.
  2203→* <mark style="color:red;">`3`</mark> - <mark style="color:red;">`POST_ONLY`</mark>: Only add liquidity, reject if would take liquidity.
  2204→
  2205→*Example:*
  2206→
  2207→```python
  2208→from nado_protocol.utils.appendix import build_appendix
  2209→
  2210→# Post-only order that only provides liquidity
  2211→appendix = build_appendix(
  2212→    order_type=OrderType.POST_ONLY
  2213→)
  2214→```
  2215→
  2216→### <mark style="color:red;">Reduce Only</mark>
  2217→
  2218→**1-bit (11)**. Restricts order to only decrease existing positions. Prevent accidentally increasing position size. Order will be rejected if it would increase the position in the same direction.
  2219→
  2220→*Use Cases:*
  2221→
  2222→* Risk management when closing positions.
  2223→* Taking profits without adding exposure.
  2224→* Automated position reduction strategies.
  2225→
  2226→*Example:*
  2227→
  2228→```python
  2229→from nado_protocol.utils.appendix import build_appendix
  2230→
  2231→# Reduce-only order to close part of existing position
  2232→appendix = build_appendix(
  2233→    order_type=OrderType.DEFAULT,
  2234→    reduce_only=True
  2235→)
  2236→```
  2237→
  2238→### <mark style="color:red;">Trigger Type</mark>
  2239→
  2240→**2-bits (12-13)**. Conditional execution behavior.
  2241→
  2242→*Values:*
  2243→
  2244→* <mark style="color:red;">`0`</mark> - <mark style="color:red;">`NONE`</mark>: Execute immediately (regular order).
  2245→* <mark style="color:red;">`1`</mark> - <mark style="color:red;">`PRICE`</mark>: Price-based conditional order.
  2246→* <mark style="color:red;">`2`</mark> - <mark style="color:red;">`TWAP`</mark>: Time-Weighted Average Price execution.
  2247→* <mark style="color:red;">`3`</mark> - <mark style="color:red;">`TWAP_CUSTOM_AMOUNTS`</mark>: TWAP with randomized amounts.
  2248→
  2249→*Example:*
  2250→
  2251→```python
  2252→from nado_protocol.utils.appendix import build_appendix
  2253→
  2254→# TWAP order executing 5 times with 0.5% max slippage
  2255→appendix = build_appendix(
  2256→    order_type=OrderType.DEFAULT,
  2257→    trigger_type=OrderAppendixTriggerType.TWAP,
  2258→    twap_times=5,
  2259→    twap_slippage_frac=0.005  # 0.5%
  2260→)
  2261→```
  2262→
  2263→### <mark style="color:red;">Reserved</mark>
  2264→
  2265→**50-bits (14-63)**. Reserved for future protocol extensions. Must be set to `0`.
  2266→
  2267→### <mark style="color:red;">Value</mark>
  2268→
  2269→**64-bits (64-127)**. Context-dependent data based on other flags.
  2270→
  2271→#### <mark style="color:red;">**TWAP Configuration (when trigger = 2 or 3)**</mark>
  2272→
  2273→Encodes TWAP execution parameters in the 64-bit value field:
  2274→
  2275→```json
  2276→| times  | slippage_x6 |
  2277→| 32 bits|   32 bits   |
  2278→```
  2279→
  2280→**Fields:**
  2281→
  2282→* <mark style="color:red;">`times`</mark>: Number of TWAP executions.
  2283→* <mark style="color:red;">`slippage_x6`</mark>: Maximum slippage × 1\_000\_000 (6 decimal precision).
  2284→
  2285→**Example:**
  2286→
  2287→```python
  2288→from nado_protocol.utils.appendix import build_appendix, order_twap_data
  2289→
  2290→# TWAP: 10 executions, 1% max slippage
  2291→appendix = build_appendix(
  2292→    order_type=OrderType.DEFAULT,
  2293→    trigger_type=OrderAppendixTriggerType.TWAP,
  2294→    twap_times=10,
  2295→    twap_slippage_frac=0.01  # 1%
  2296→)
  2297→
  2298→# Extract TWAP data
  2299→times, slippage = order_twap_data(appendix)
  2300→# times = 10, slippage = 0.01
  2301→```
  2302→
  2303→#### <mark style="color:red;">**Isolated Margin (when isolated = 1)**</mark>
  2304→
  2305→Amount of quote (margin\_x6) to transfer to isolated subaccount on first fill, stored in the 64-bit value field.
  2306→
  2307→{% hint style="warning" %}
  2308→**Important:** Isolated margin is stored in **x6 precision** (6 decimals) in the appendix value field.
  2309→
  2310→* Stored as `margin_x6` (6 decimal places)
  2311→* Takes up 64 bits (bits 64-127 of the appendix)
  2312→  {% endhint %}
  2313→
  2314→*Example:*
  2315→
  2316→```python
  2317→from nado_protocol.utils.appendix import build_appendix, order_isolated_margin
  2318→from nado_protocol.utils.math import to_x6
  2319→
  2320→# Isolated order with 500 USDT0 margin
  2321→appendix = build_appendix(
  2322→    order_type=OrderType.DEFAULT,
  2323→    isolated=True,
  2324→    isolated_margin=to_x6(500)  # 500 USDT0 (x6 precision)
  2325→)
  2326→
  2327→# Extract isolated margin
  2328→margin = order_isolated_margin(appendix)
  2329→# Returns: 500000000 (500 * 10^6 in x6 precision)
  2330→```
  2331→
  2332→## Constraints
  2333→
  2334→* **Isolated + TWAP**: Cannot combine isolated orders with TWAP (trigger types 2 or 3).
  2335→* **TWAP Requirements**: TWAP orders must specify both <mark style="color:red;">`twap_times`</mark> and <mark style="color:red;">`twap_slippage_frac`</mark> .
  2336→* **Isolated Margin**: Can only set <mark style="color:red;">`isolated_margin`</mark> when <mark style="color:red;">`isolated=True`</mark> .
  2337→
  2338→## Migration from Legacy Format
  2339→
  2340→**Before (deprecated):**
  2341→
  2342→* Order type encoded in <mark style="color:red;">`expiration`</mark> field.
  2343→* Reduce-only flag encoded in <mark style="color:red;">`nonce`</mark> field.
  2344→* Limited trigger functionality.
  2345→
  2346→**After (current):**
  2347→
  2348→* All flags consolidated in 128-bit <mark style="color:red;">`appendix`</mark> .
  2349→* <mark style="color:red;">`expiration`</mark> is pure timestamp.
  2350→* <mark style="color:red;">`nonce`</mark> encodes <mark style="color:red;">`recv_time`</mark> only.
  2351→* Enhanced trigger and isolated margin support.
  2352→

  2386→#### Manual Bit Manipulation (Advanced)

  2393→# Build appendix manually
  2394→def build_manual_appendix(order_type=0, isolated=False, reduce_only=False,
  2395→                         trigger_type=0, value=0):
